🦀 Tauri란 무엇인가?

Tauri는 Rust로 작성된 크로스플랫폼 데스크탑 앱 프레임워크입니다. 프론트엔드는 웹 기술(HTML/CSS/JS + 모든 프레임워크)로 구성하고, 백엔드(네이티브 기능)는 Rust로 처리합니다. Electron과 동일한 목표를 가지지만, 번들 크기는 10배 이상 작고, 메모리 사용량도 현저히 낮습니다.

┌─────────────────────────────────────────────────────────┐ │ Tauri Application │ │ │ │ ┌─────────────────────┐ ┌─────────────────────────┐│ │ │ Frontend (UI) │ │ Backend (Rust) ││ │ │ │ │ ││ │ │ React / Vue / │◄──► Commands (함수 노출) ││ │ │ Svelte / Vanilla │ │ State 관리 ││ │ │ │ │ OS API 접근 ││ │ │ (WebView 렌더링) │ │ 파일시스템, 네트워크 ││ │ └─────────────────────┘ └─────────────────────────┘│ │ │ │ │ │ IPC Bridge (invoke / emit) │ │ │ ▼ │ │ ┌───────────────────────┐ │ │ │ OS Native APIs │ │ │ │ (Windows/macOS/Linux)│ │ │ └───────────────────────┘ │ └─────────────────────────────────────────────────────────┘
백엔드 개발자 관점: Tauri의 Rust 백엔드는 REST API 서버와 유사합니다. 프론트엔드에서 invoke('command_name', args)를 호출하면 Rust 함수가 실행되는 구조 — HTTP 요청 대신 IPC를 사용하는 것이라고 이해하면 됩니다.

🏗️ 아키텍처 구조 심층 이해
이론

핵심 구성 요소

  • tauri (Rust crate) — 메인 런타임. 앱 생명주기, 윈도우 관리, IPC 처리
  • tauri-build — 빌드 시 리소스 번들링, 아이콘 처리
  • @tauri-apps/api (JS) — 프론트엔드에서 Rust 기능을 호출하는 JS 바인딩
  • Vite / Webpack — 프론트엔드 번들러 (Tauri는 번들러에 무관)

IPC (Inter-Process Communication) 동작 원리

Frontend (JS) Backend (Rust) │ │ │ invoke("get_user", {id}) │ │ ─────────────────────────► │ │ │ #[tauri::command] │ │ fn get_user(id: u32) -> User { ... } │ │ │ Promise ◄──────── │ return User { ... } │ │

프로세스 모델

  • Main Process (Rust): OS 리소스 접근, 윈도우 생성, 비즈니스 로직
  • Renderer Process (WebView): UI 렌더링 — JS/HTML/CSS 실행
  • 둘 사이의 통신은 IPC Bridge를 통해 직렬화된 메시지로 처리
백엔드 관점 비유: Main Process = API 서버, Renderer = 클라이언트 앱, invoke() = HTTP POST 요청, Command = Controller/Handler 함수.

📡 Commands — IPC의 핵심
이론 실무

Commands는 Tauri의 핵심 IPC 메커니즘입니다. Rust 함수에 #[tauri::command] 어노테이션을 붙이면 JS에서 호출 가능한 "엔드포인트"가 됩니다.

Rust — Command 정의

// src-tauri/src/lib.rs
use tauri::Manager;

// 간단한 동기 Command
#[tauri::command]
fn greet(name: &str) -> String {
    format!("Hello, {}! You've been greeted from Rust!", name)
}

// 비동기 Command (I/O, DB, 외부 API 호출 등)
#[tauri::command]
async fn fetch_data(url: String) -> Result<String, String> {
    reqwest::get(&url)
        .await
        .map_err(|e| e.to_string())?
        .text()
        .await
        .map_err(|e| e.to_string())
}

// Command 등록
#[cfg_attr(mobile, tauri::mobile_entry_point)]
pub fn run() {
    tauri::Builder::default()
        .invoke_handler(tauri::generate_handler![greet, fetch_data])
        .run(tauri::generate_context!())
        .expect("error while running tauri application");
}

JavaScript — Command 호출

// src/App.tsx
import { invoke } from '@tauri-apps/api/core';

// 기본 호출
const message = await invoke<string>('greet', { name: 'World' });

// 에러 처리
try {
  const data = await invoke<string>('fetch_data', { url: 'https://api.example.com' });
  console.log(data);
} catch (err) {
  console.error('Rust error:', err);  // Rust의 Err(String) 값
}

복잡한 타입 전달 (Serde)

// Rust — 구조체 직렬화
use serde::{Deserialize, Serialize};

#[derive(Serialize, Deserialize)]
struct User {
    id: u32,
    name: String,
    email: String,
}

#[tauri::command]
fn get_user(id: u32) -> Result<User, String> {
    // DB 조회 등...
    Ok(User { id, name: "Alice".into(), email: "alice@example.com".into() })
}
실무 팁: Command 이름은 snake_case (Rust 관례). JS에서 호출 시 camelCase로도 쓸 수 있도록 convert_case를 쓰거나, 그냥 snake_case 그대로 사용합니다. Tauri 2.0에서는 기본적으로 camelCase 변환을 지원합니다.
📢 Events 시스템
이론 실무

Commands가 요청-응답(Request-Response) 방식이라면, Events는 단방향 메시지 브로드캐스트입니다. 백그라운드 작업 진행률, 실시간 데이터 스트리밍 등에 활용합니다.

Rust → Frontend (이벤트 발행)

#[tauri::command]
async fn long_task(app: tauri::AppHandle) {
    for i in 0..100 {
        // 진행률 이벤트 발행
        app.emit("progress", i).unwrap();
        tokio::time::sleep(Duration::from_millis(50)).await;
    }
    app.emit("task-complete", "done").unwrap();
}

Frontend → Rust (이벤트 수신)

import { listen } from '@tauri-apps/api/event';

// 이벤트 구독
const unlisten = await listen<number>('progress', (event) => {
    setProgress(event.payload);   // event.payload = 진행률
});

// 컴포넌트 언마운트 시 구독 해제 (메모리 누수 방지)
return () => unlisten();

상황CommandsEvents
단순 데이터 조회✅ 적합-
비즈니스 로직 실행✅ 적합-
파일 다운로드 진행률-✅ 적합
실시간 로그 스트리밍-✅ 적합
OS 알림 (트레이 클릭)-✅ 적합

초급 면접 질문

Q. Tauri와 Electron의 가장 큰 차이점은 무엇인가요?
A. 가장 큰 차이는 WebView 접근 방식입니다. Electron은 Chromium을 앱에 직접 번들하여 일관된 렌더링을 보장하지만 앱 크기가 100MB 이상이 됩니다. Tauri는 OS 기본 WebView(WKWebView/WebView2/WebKitGTK)를 사용하여 앱 크기를 5–15MB로 줄이고 메모리 사용량도 낮춥니다. 또한 백엔드를 Node.js 대신 Rust로 작성하여 성능과 보안이 향상됩니다.
Q. Tauri에서 프론트엔드와 백엔드가 통신하는 방법은?
A. IPC(Inter-Process Communication) 방식으로 통신합니다. 두 가지 방법이 있습니다: (1) Commands — JS에서 invoke('command_name', args)를 호출하면 Rust의 #[tauri::command] 함수가 실행되어 결과를 Promise로 반환합니다. (2) Events — Rust에서 app.emit()으로 이벤트를 발행하면 JS에서 listen()으로 수신합니다. 단방향 스트리밍 시나리오(진행률, 실시간 데이터)에 적합합니다.
Q. Tauri 앱에서 Rust를 왜 사용하나요? Node.js를 쓰면 안 되나요?
A. Tauri의 백엔드는 Rust로 고정되어 있습니다. Rust를 선택한 이유는 메모리 안전성(GC 없이 메모리 오류 방지), 네이티브 성능, 작은 바이너리 크기 때문입니다. 단, 백엔드 비즈니스 로직 일부를 Node.js로 처리하고 싶다면 Tauri 앱 내에서 sidecar(사이드카) 프로세스로 Node.js 서버를 실행하는 방법이 있습니다.
Q. tauri.conf.json의 역할은 무엇인가요?
A. 앱의 메타데이터(이름, 버전, 번들 ID)부터 윈도우 크기, 보안 정책(CSP, 권한 허용 목록), 빌드 설정, 업데이트 서버 설정까지 모든 Tauri 설정을 담는 중앙 설정 파일입니다. package.json + 프레임워크 설정 + 앱 매니페스트를 합쳐 놓은 것으로 이해하면 됩니다.

🗄️ 상태 관리 (State Management)
이론 실무

Tauri는 앱 전역 상태를 Rust 타입 시스템으로 안전하게 관리합니다. manage()로 상태를 등록하고, Command 파라미터로 State<T>를 받아 접근합니다.

기본 State 사용

use std::sync::Mutex;
use tauri::State;

// 공유 상태 타입
struct AppState {
    counter: Mutex<u32>,
    user_name: Mutex<String>,
}

#[tauri::command]
fn increment(state: State<AppState>) -> u32 {
    let mut counter = state.counter.lock().unwrap();
    *counter += 1;
    *counter
}

// 앱 시작 시 상태 등록
pub fn run() {
    tauri::Builder::default()
        .manage(AppState {
            counter: Mutex::new(0),
            user_name: Mutex::new(String::new()),
        })
        .invoke_handler(tauri::generate_handler![increment])
        .run(tauri::generate_context!())
        .unwrap();
}

DB 연동 실무 패턴

use sqlx::SqlitePool;

struct DbState(SqlitePool);

#[tauri::command]
async fn get_users(state: State<'_, DbState>) -> Result<Vec<User>, String> {
    sqlx::query_as!(User, "SELECT * FROM users")
        .fetch_all(&state.0)
        .await
        .map_err(|e| e.to_string())
}

// 앱 시작 시 DB 초기화
#[cfg_attr(mobile, tauri::mobile_entry_point)]
pub fn run() {
    tauri::async_runtime::block_on(async {
        let pool = SqlitePool::connect("sqlite:app.db").await.unwrap();
        tauri::Builder::default()
            .manage(DbState(pool))
            ...
    });
}
Mutex vs RwLock: 읽기 비중이 높다면 RwLock이 성능상 유리합니다. 단순한 경우 Mutex가 더 안전합니다.
📁 파일시스템 & 다이얼로그
이론 실무

경로 API (app_data_dir 등)

// Rust — 안전한 경로 접근
use tauri::Manager;

#[tauri::command]
fn get_app_dir(app: tauri::AppHandle) -> String {
    app.path()
        .app_data_dir()
        .unwrap()
        .to_string_lossy()
        .to_string()
    // macOS: ~/Library/Application Support/com.example.app
    // Windows: %APPDATA%\com.example.app
    // Linux: ~/.local/share/com.example.app
}

파일 읽기/쓰기

#[tauri::command]
async fn write_settings(
    app: tauri::AppHandle,
    content: String,
) -> Result<(), String> {
    let mut path = app.path().app_data_dir().unwrap();
    path.push("settings.json");
    std::fs::write(path, content).map_err(|e| e.to_string())
}

파일 선택 다이얼로그

// JS — 파일 열기 다이얼로그
import { open, save } from '@tauri-apps/plugin-dialog';

// 파일 열기
const filePath = await open({
    multiple: false,
    filters: [{ name: 'JSON', extensions: ['json'] }]
});

// 저장 다이얼로그
const savePath = await save({
    filters: [{ name: 'Text', extensions: ['txt'] }],
    defaultPath: '~/Desktop/output.txt'
});
보안 주의: Tauri 1.x에서는 allowlist.fs에서 접근 허용 경로를 명시적으로 설정해야 합니다. Tauri 2.0에서는 capabilities JSON으로 권한을 선언합니다. 임의 경로 접근은 기본적으로 차단됩니다.
🌐 HTTP 클라이언트 & 외부 API 호출
이론 실무

Tauri 앱에서 외부 HTTP 요청은 프론트엔드(fetch)와 Rust 백엔드(reqwest) 두 가지 방법으로 처리할 수 있습니다.

방법장점단점적합한 경우
프론트엔드 fetch 간단, 익숙한 방식 CSP 제한, CORS 문제 단순 공개 API 호출
Rust reqwest CORS 우회, API 키 보호, 고성능 Rust 코드 필요 API 키 포함 요청, 대용량 다운로드

Rust로 외부 API 호출 (API 키 보호)

// Cargo.toml
[dependencies]
reqwest = { version = "0.12", features = ["json"] }
serde_json = "1.0"

// lib.rs
#[tauri::command]
async fn call_openai(prompt: String) -> Result<String, String> {
    let api_key = std::env::var("OPENAI_API_KEY")
        .map_err(|_| "API key not set".to_string())?;

    let client = reqwest::Client::new();
    let response = client
        .post("https://api.openai.com/v1/chat/completions")
        .bearer_auth(api_key)
        .json(&serde_json::json!({
            "model": "gpt-4",
            "messages": [{"role": "user", "content": prompt}]
        }))
        .send().await
        .map_err(|e| e.to_string())?
        .text().await
        .map_err(|e| e.to_string())?;

    Ok(response)
}
// API 키가 Rust 코드 안에만 있어 JS에 노출되지 않음

Tauri 보안 철학: Least Privilege (최소 권한)

Tauri는 기본적으로 모든 기능을 차단하고 필요한 것만 명시적으로 허용하는 화이트리스트 모델을 따릅니다.

// src-tauri/capabilities/default.json
{
  "identifier": "default",
  "description": "기본 권한 세트",
  "windows": ["main"],
  "permissions": [
    "core:default",
    "fs:allow-read-text-file",  // 텍스트 파일 읽기만 허용
    "fs:allow-write-text-file", // 텍스트 파일 쓰기만 허용
    "dialog:allow-open",        // 파일 열기 다이얼로그만 허용
    "shell:allow-open"          // URL을 기본 브라우저로 열기만 허용
  ]
}

중급 면접 질문

Q. Tauri에서 API 키를 안전하게 관리하는 방법은?
A. API 키를 프론트엔드(JS)에 절대 노출하면 안 됩니다. 대신 Rust Command 안에서 환경 변수(std::env::var) 또는 OS 키체인(tauri-plugin-keystore)에서 읽어 사용합니다. 빌드 시점에 환경 변수로 주입한 경우에도 바이너리 문자열 검색으로 노출될 수 있으므로, 런타임에 키체인/환경 변수에서 로드하는 방식이 가장 안전합니다.
Q. Tauri의 Capabilities 시스템(2.0)과 Allowlist(1.x)의 차이점은?
A. Tauri 1.x의 allowlist는 tauri.conf.json에 boolean 플래그로 기능을 on/off하는 단순한 방식이었습니다. Tauri 2.0의 Capabilities 시스템은 윈도우별로 독립적인 권한 세트를 선언할 수 있어 더 세밀한 권한 제어가 가능합니다. 예를 들어 main 윈도우와 settings 윈도우에 서로 다른 파일 접근 권한을 부여할 수 있습니다. 또한 원격 도메인별 권한도 설정 가능합니다.
Q. Tauri에서 SQLite를 사용할 때 고려해야 할 사항은?
A. (1) DB 파일 경로는 반드시 app_data_dir() 같은 OS 표준 앱 데이터 경로에 저장해야 합니다. (2) 멀티스레드 환경에서 Connection Pool(sqlx의 SqlitePool)을 State로 관리해야 합니다. (3) 앱 배포 후 스키마 변경은 마이그레이션 도구(sqlx-cli의 migrate 기능)로 처리해야 합니다. (4) DB 암호화가 필요하면 SQLCipher를 사용합니다.
Q. macOS 배포 시 Notarization이 필요한 이유는?
A. Apple의 Gatekeeper 보안 기능 때문입니다. macOS Catalina(10.15)부터 Apple Notarization을 받지 않은 앱은 "악성 소프트웨어가 포함되어 있어 열 수 없습니다" 경고가 표시됩니다. Notarization은 Apple 서버에 앱을 업로드해 악성코드 검사를 받고 공증 티켓을 받는 과정입니다. tauri-action GitHub Action을 사용하면 자동화할 수 있습니다.

// my-plugin/src/lib.rs
use tauri::{
    plugin::{Builder, TauriPlugin},
    Manager, Runtime,
};

pub struct MyPlugin {
    config: PluginConfig,
}

impl MyPlugin {
    pub fn new(config: PluginConfig) -> Self {
        Self { config }
    }
}

#[tauri::command]
async fn plugin_action<R: Runtime>(
    app: tauri::AppHandle<R>,
    param: String,
) -> Result<String, String> {
    // 플러그인 로직
    Ok(format!("processed: {}", param))
}

pub fn init<R: Runtime>(config: PluginConfig) -> TauriPlugin<R> {
    Builder::<R>::new("my-plugin")
        .invoke_handler(tauri::generate_handler![plugin_action])
        .setup(move |app, _api| {
            app.manage(MyPlugin::new(config.clone()));
            Ok(())
        })
        .build()
}

// 앱에서 사용
tauri::Builder::default()
    .plugin(my_plugin::init(PluginConfig { ... }))
    ...

use tauri::{WebviewUrl, WebviewWindowBuilder};

#[tauri::command]
async fn open_settings(app: tauri::AppHandle) {
    // 이미 열려있으면 포커스, 없으면 새로 생성
    if let Some(win) = app.get_webview_window("settings") {
        win.set_focus().unwrap();
        return;
    }
    WebviewWindowBuilder::new(&app, "settings", WebviewUrl::App("/settings".into()))
        .title("Settings")
        .inner_size(600.0, 500.0)
        .resizable(false)
        .build()
        .unwrap();
}

// 윈도우 간 통신
#[tauri::command]
fn send_to_main(app: tauri::AppHandle, data: String) {
    if let Some(main) = app.get_webview_window("main") {
        main.emit("from-settings", data).unwrap();
    }
}

시스템 트레이 (System Tray)

use tauri::{
    menu::{Menu, MenuItem},
    tray::{TrayIcon, TrayIconBuilder},
};

fn setup_tray(app: &mut tauri::App) -> Result<(), Box<dyn std::error::Error>> {
    let show = MenuItem::with_id(app, "show", "Show", true, None::<&str>)?;
    let quit = MenuItem::with_id(app, "quit", "Quit", true, None::<&str>)?;
    let menu = Menu::with_items(app, &[&show, &quit])?;

    TrayIconBuilder::new()
        .icon(app.default_window_icon().unwrap().clone())
        .menu(&menu)
        .on_menu_event(move |app, event| match event.id().as_ref() {
            "show" => {
                app.get_webview_window("main").unwrap().show().unwrap();
            }
            "quit" => app.exit(0),
            _ => {}
        })
        .build(app)?;
    Ok(())
}

// 앱 종료 시 트레이로 숨기기
tauri::Builder::default()
    .setup(|app| { setup_tray(app)?; Ok(()) })
    .on_window_event(|window, event| {
        if let tauri::WindowEvent::CloseRequested { api, .. } = event {
            window.hide().unwrap();
            api.prevent_close();  // 실제 종료 방지
        }
    })

}

#[tauri::command]
fn read_user_file(
    app: tauri::AppHandle,
    filename: String,
) -> Result<String, String> {
    // Path traversal 방어: ../../../etc/passwd 같은 패턴 차단
    if filename.contains("..") || filename.contains("/") {
        return Err("invalid filename".to_string());
    }
    let mut path = app.path().app_data_dir().unwrap();
    path.push(&filename);
    std::fs::read_to_string(path).map_err(|e| e.to_string())
}

고급 면접 질문

Q. Tauri에서 메모리 안전성이 어떻게 보장되나요?
A. Rust의 소유권(Ownership) 시스템이 컴파일 타임에 메모리 오류(use-after-free, double-free, buffer overflow)를 방지합니다. GC(가비지 컬렉터) 없이도 메모리를 안전하게 관리하므로 런타임 오버헤드가 없습니다. Mutex<T>, Arc<T> 등의 타입으로 멀티스레드 공유도 타입 시스템에서 안전성을 보장합니다. 이것이 Electron(Node.js의 V8 GC 의존)과의 근본적인 차이입니다.
Q. Tauri 앱을 역방향 엔지니어링(리버싱)으로부터 어떻게 보호하나요?
A. (1) Rust 바이너리는 LLVM 최적화로 역어셈블리가 어렵습니다. strip = true로 심볼 제거. (2) 프론트엔드 JS는 WebView에서 실행되므로 DevTools 접근을 production 빌드에서 비활성화해야 합니다(devtools = false). (3) 민감한 비즈니스 로직은 반드시 Rust Command로 이동 — JS 코드는 항상 추출 가능하다고 가정해야 합니다. (4) API 키 등 시크릿은 OS 키체인 또는 서버에 보관.
Q. dangerousRemoteDomainIpcAccess를 절대 사용하면 안 되는 이유는?
A. 이 옵션을 활성화하면 WebView가 로드하는 원격 도메인의 JS가 Tauri Command(IPC)를 호출할 수 있게 됩니다. 원격 도메인이 해킹당하거나 CDN이 오염되면, 공격자의 JS가 파일시스템 쓰기, 쉘 실행 등 허용된 모든 Command를 실행할 수 있습니다. Tauri 앱은 항상 로컬 에셋(embedded assets)만 서비스하고, 원격 도메인이 필요한 데이터는 Command를 통해 Rust에서 fetch 후 반환하는 패턴을 사용해야 합니다.
Q. Tauri 2.0에서 멀티 WebView를 어떤 시나리오에서 사용하나요?
A. 하나의 네이티브 윈도우 안에 여러 WebView를 분할 배치하는 기능입니다. 예: (1) IDE 스타일 앱에서 에디터 영역과 프리뷰 영역을 완전히 독립된 WebView로 격리 — 프리뷰가 크래시해도 에디터에 영향 없음. (2) 다국어 앱에서 마케팅 사이트(원격)와 앱 UI(로컬)를 동일 윈도우에 표시하되 서로 다른 CSP 적용. (3) 보안 레벨이 다른 두 영역을 격리.
Q. Tauri로 대용량 파일(수 GB) 처리를 최적화하는 방법은?
A. (1) 파일 경로만 IPC로 전달하고 Rust에서 직접 처리 — JS로 데이터를 주고받지 않음. (2) Rust의 BufReader로 스트리밍 처리. (3) 진행률은 Events로 주기적으로 프론트엔드에 push. (4) 사용자 취소를 위해 tokio::sync::CancellationToken 사용. (5) 메모리 매핑(memmap2 crate)으로 OS 레벨 스트리밍 활용.
INTERVIEW

면접 질문 총정리

🎯 초급 핵심 질문
Q1. Tauri 앱의 번들 크기가 Electron보다 작은 이유는?
OS에 이미 설치된 WebView(WKWebView/WebView2)를 사용하기 때문에 Chromium을 번들할 필요가 없습니다. Rust 바이너리도 GC와 런타임이 없어 Node.js보다 작습니다.
Q2. invoke()와 listen()의 차이점은?
invoke()는 요청-응답 패턴(Promise 반환). listen()은 Rust에서 발행하는 이벤트를 구독하는 단방향 스트림 패턴. 진행률, 실시간 알림 등에 listen() 사용.
Q3. #[tauri::command] 어노테이션의 역할은?
Rust 함수를 IPC를 통해 JS에서 호출 가능한 Command로 노출합니다. 함수의 파라미터/반환값은 Serde JSON으로 자동 직렬화/역직렬화됩니다.
Q4. Tauri 개발 환경에서 Hot Reload가 동작하는 방식은?
npm run tauri dev 실행 시 Vite dev server가 프론트엔드 HMR을 제공하고, Rust 소스 변경 시 cargo가 재컴파일 후 앱을 재시작합니다. 프론트엔드 변경은 즉시 반영, Rust 변경은 재컴파일 후 반영.
🎯 중급 핵심 질문
Q5. Tauri State의 Mutex가 필요한 이유는?
async Command들이 여러 스레드에서 동시에 실행될 수 있기 때문입니다. Rust의 소유권 규칙상 공유 가변 참조는 Mutex(또는 RwLock)로 보호해야 컴파일됩니다. 이를 통해 데이터 레이스를 컴파일 타임에 방지합니다.
Q6. Tauri 앱에서 SQLite를 사용하는 일반적인 패턴은?
SqlitePool을 State로 manage()에 등록 → async Command에서 State<DbState>로 주입 → sqlx::query_as!() 매크로로 타입 안전 쿼리 실행. DB 파일은 app_data_dir()에 저장.
Q7. Content Security Policy(CSP)를 Tauri에서 설정해야 하는 이유는?
XSS 공격으로 악성 스크립트가 주입되어 invoke()를 호출하는 것을 방지하기 위해서입니다. Tauri 앱은 로컬 파일 접근 등 강력한 권한을 가지므로 XSS의 파급력이 일반 웹보다 훨씬 큽니다.
Q8. tauri-plugin-store vs 직접 파일 IO의 차이점은?
tauri-plugin-store는 JSON 파일 기반 키-값 저장소로 쓰기 최적화(디바운스), 타입 안전 API, JS/Rust 양쪽에서 동일 인터페이스 제공. 직접 파일 IO는 더 유연하지만 직렬화, 동시성, 경로 관리를 직접 처리해야 합니다.
🎯 고급 핵심 질문
Q9. Tauri 커스텀 플러그인과 일반 Command의 차이는?
플러그인은 독립 Rust crate로 setup 훅, 상태 관리, JS 바인딩 패키지를 포함한 완전한 기능 단위입니다. 여러 앱에서 재사용 가능하고 npm/crates.io에 배포할 수 있습니다. 일반 Command는 하나의 앱 내부 함수입니다.
Q10. Tauri 업데이터의 Ed25519 서명이 왜 중요한가요?
업데이트 파일의 무결성과 출처를 보장합니다. 업데이트 서버가 해킹되어 악성 바이너리가 업로드되더라도, 개인키로 서명되지 않은 파일은 설치가 거부됩니다. MITM 공격도 방어합니다. 개인키를 잃으면 이후 업데이트가 불가능하므로 안전한 백업이 필수입니다.
Q11. Tauri 2.0의 Capabilities 시스템에서 윈도우별 권한 분리의 실용적 예시는?
예: 메인 윈도우는 파일 읽기/쓰기 모두 허용, 외부 URL을 로드하는 iframe/보조 윈도우는 파일시스템 접근 차단. 또는 관리자 윈도우에만 DB 쓰기 권한 부여, 일반 사용자 윈도우는 읽기만 허용.
Q12. Tauri 앱에서 긴 작업(백그라운드 처리)을 구현하는 패턴은?
async Command에서 tokio::spawn으로 백그라운드 태스크 실행 → AppHandle.emit()으로 진행률 이벤트 발행 → 취소는 Arc<Mutex<bool>> 또는 CancellationToken으로 구현 → JS에서 listen()으로 진행률 구독, 취소 버튼은 invoke()로 취소 플래그 설정.
Q13. Tauri vs Flutter Desktop 선택 기준은?
Tauri: 기존 웹 스택(React/Vue 등) 재사용, 웹 개발 팀, 번들 크기 최소화, 보안 민감 앱. Flutter: 동일 코드로 모바일+데스크탑 지원, 일관된 커스텀 렌더링, 네이티브 WebView에 의존하지 않는 완전한 커스텀 UI. 웹 기반 UI를 데스크탑으로 포팅한다면 Tauri, 완전히 새로운 크로스플랫폼 앱이라면 Flutter도 고려.
Q14. Tauri의 sidecar 기능은 무엇이고 어떤 경우에 사용하나요?
Sidecar는 Tauri 앱과 함께 번들된 외부 실행 파일(바이너리)을 실행하는 기능입니다. 예: Python 스크립트, Node.js 서버, Go 바이너리. Rust로 포팅하기 어려운 기존 비즈니스 로직을 그대로 유지하면서 Tauri 앱에서 제어할 때 사용합니다. IPC는 stdin/stdout 또는 로컬 소켓으로 처리합니다.
